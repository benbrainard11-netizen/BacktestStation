# 247 prompt — dataset snapshots + run provenance

_2026-05-17 PM / benpc → ben-247._

## TL;DR

The trial registry you just shipped (`d910324`) gave us protocol lock chains. But `trial_lock_records.dataset_snapshot_id` is currently a free-form string with no enforcement. Three reviewers in a row have flagged the same gap: **no run can prove what data + code produced it.** Time to close that gap.

Build a structured `dataset_snapshots` table + add provenance columns to `backtest_runs`. ~1-2 days. Single PR. After this, every locked walk-forward + every backtest can prove its full lineage.

## What v20 just made obvious

v20 (results: `experiments/locked_walkforward_2026_05_17/results/`) ran fine but: the lockfile referenced `research_events_manifest_sha256: "5ad286d2..."` as a free-form string. There's no table where that hash is durable. If `data/research_events/manifest.json` changes, no DB row reflects it. The next reviewer to ask "what data did this run use?" gets a YAML file, not a database fact.

5.5 Pro's review explicitly listed this:

> "Every run must have a snapshot. Without that, the database can grow while trust does not."

## What to build

### 1. `dataset_snapshots` table

```sql
CREATE TABLE dataset_snapshots (
    id INTEGER PRIMARY KEY,
    -- Hash-derived stable identifier. Generated by the creation utility,
    -- not the DB autoincrement. Lets multiple PCs reference the same snapshot.
    snapshot_id VARCHAR(64) NOT NULL UNIQUE,
    name VARCHAR(200),                              -- optional human label
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by VARCHAR(80),                         -- 'benpc' / 'ben-247' / etc.
    -- The data scope
    symbols_json TEXT NOT NULL,                     -- JSON array of symbol IDs
    date_start DATETIME NOT NULL,
    date_end DATETIME NOT NULL,
    schemas_json TEXT NOT NULL,                     -- e.g., ["ohlcv-1m", "tbbo", "mbp-1"]
    -- The integrity proofs
    r2_inventory_hash VARCHAR(64),                  -- sha256 of _research_inventory.json at snapshot time
    research_events_manifest_sha256 VARCHAR(64),    -- sha256 of data/research_events/manifest.json
    partition_count INTEGER NOT NULL,
    total_bytes BIGINT,
    -- Lineage / exclusions
    roll_map_version VARCHAR(40),                   -- if applicable
    known_exclusions_json TEXT,                     -- JSON array of {date, symbol, reason}
    -- Lifecycle
    status VARCHAR(20) NOT NULL DEFAULT 'draft',    -- draft / active / archived
    notes TEXT,
    -- Optional FK to partition-level validation when that table lands
    validation_report_id INTEGER                    -- nullable, no FK constraint yet
);
CREATE INDEX idx_dataset_snapshots_status ON dataset_snapshots(status);
CREATE INDEX idx_dataset_snapshots_created_at ON dataset_snapshots(created_at);
```

Partitions:
- For v1, store the partition list as a JSON column OR a separate `dataset_snapshot_partitions(snapshot_id, r2_key, size, sha256)` table.
- Your call on which. JSON is simpler; separate table is queryable. **If you have a preference, pick it; if not, separate table for queryability.**

### 2. Add 3 columns to `backtest_runs`

```sql
ALTER TABLE backtest_runs ADD COLUMN dataset_snapshot_id VARCHAR(64);  -- FK by string to dataset_snapshots.snapshot_id
ALTER TABLE backtest_runs ADD COLUMN code_commit_sha VARCHAR(40);
ALTER TABLE backtest_runs ADD COLUMN seed INTEGER;
CREATE INDEX idx_backtest_runs_snapshot ON backtest_runs(dataset_snapshot_id);
```

Notes:
- `dataset_snapshot_id` is **nullable** for backfill compatibility. Historical runs (v8a, v13-v19) lack snapshots; that's fine, the column just stays null.
- Going forward, any locked-walk-forward or production-grade run MUST set it. Enforce via the runner code, not the DB.

### 3. Update `trial_lock_records.dataset_snapshot_id`

Currently a free-form `VARCHAR(160)`. Decide:
- (a) Leave as free-form string, but **change the semantics**: by convention it now stores the `snapshot_id` value (the hash) that matches `dataset_snapshots.snapshot_id`. No FK constraint, but query-joinable.
- (b) Add a hard FK from `trial_lock_records.dataset_snapshot_id` → `dataset_snapshots.snapshot_id`. Requires changing the column type to VARCHAR(64).

I'd vote (a) for v1 — softer coupling, easier to migrate existing free-form values later. Your call.
